## Phase Goal

**As a** cyclist or runner mid-activity, **I want to** see my elapsed time, current speed or pace, and distance floating on my glasses, updating every second, **so that** I keep my eyes on the road and my phone in my pocket while my ride or run is measured.

(Direct transliteration of the ROADMAP Phase 2 Goal + project Core Value; MVP mode, vertical-slice plan set.)

<objective>
Wire the live data path and fix the layout-mode ownership bug. Two behavioral edit sites, exactly as enumerated by RESEARCH Pitfall 2's construction-site table: (1) the Phase-1 no-op `is ParsedMessage.SportState -> { }` becomes a `currentState.applySportState(...)` copy, delivering 1Hz HUD updates through the existing `onStateUpdate` plumbing; (2) `toggleLayout()` routes through BluetoothClient so `currentState` — the replicated source of truth that every incoming message re-projects onto `hudView.state` — carries the tapped mode.

Purpose: Without edit (2), HUD-03 fails on device exactly when it matters: during recording, sport_state arrives at 1Hz and any tap-selected mode visibly reverts within a second (verified code path: HudActivity.kt:89-95 lambda wholesale-replaces hudView.state from currentState; BluetoothClient.kt:263 fires onStateUpdate after EVERY message). This fix is load-bearing for the whole phase.
Output: Live sport fields in HudState at 1Hz; a tap cycle that survives streaming.

MVP slice: after this plan, tapping to SPORT during a recording sticks, and the (still minimal) SPORT screen is fed real 1Hz data — full rendering lands in Wave 3.
</objective>
