    Per RESEARCH Code Examples §3 (adopt as written — the in-View variant auto-scopes to the view lifecycle and needs no Activity plumbing):
    1. Add a `private val sportTick = object : Runnable` whose run() checks `state.layoutMode == MapLayoutMode.SPORT`, and if so calls `invalidate()` then `postDelayed(this, 1000L)` — the self-rescheduling shape of the phone's sportStateTicker and HudActivity's exitWhenStoppedRunnable. The layoutMode guard makes over-scheduling harmless and stops the chain automatically if the mode changed between ticks. NO coroutines, NO Timer, NO thread-sleep loop (locked constraint: Handler-family scheduling only).
    2. Replace the state setter body (:99-100) with the transition-edge form: compute enteredSport/leftSport by comparing value.layoutMode against field.layoutMode before assignment; after `field = value`: if enteredSport { removeCallbacks(sportTick); postDelayed(sportTick, 1000L) }; if leftSport removeCallbacks(sportTick); always postInvalidate() (existing behavior preserved for every mode).
    3. Override onAttachedToWindow(): call super, then if state.layoutMode == SPORT { removeCallbacks(sportTick); postDelayed(sportTick, 1000L) } — belt-and-braces re-arm (RESEARCH A2).
    4. Override onDetachedFromWindow(): removeCallbacks(sportTick) BEFORE super — guarantees no leaked callbacks when the view is torn down.
    Why this exists: without the ticker, redraws only happen on message arrival — so when messages STOP (the exact staleness scenario), the >3s dim and >10s NO DATA transitions would never appear on screen (HUD-02 staleness UX; device-proven in 02-04 step 7).
  </action>
